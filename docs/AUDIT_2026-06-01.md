## CRITICAL

### connectionStateMutex 비재진입 데드락 — 자동 재연결이 영구 멈춤

- **파일**: `app/src/main/java/com/inik/camcon/data/datasource/ptpip/PtpipDataSource.kt:368-430, 507-528, 691-692`
- **분류**: race | **차원**: PTP-IP 네트워크 | **확신**: high

**문제**
handleNetworkStateChange는 connectionStateMutex.withLock(370)을 잡은 상태에서 attemptAutoReconnect(405, 423)를 호출하고, attemptAutoReconnect는 다시 connectToCamera(528)를 호출하는데 connectToCamera 본체는 connectionStateMutex.withLock(692)으로 같은 Mutex를 재획득하려 한다. kotlinx Mutex는 재진입(reentrant)이 아니므로 동일 코루틴이 이미 보유한 락을 다시 withLock하면 영구 대기에 빠진다. 즉 Wi-Fi 재연결/AP IP 변경 자동 재연결 분기가 진입하면 그 코루틴은 데드락되어 다시는 완료되지 않는다. _connectionState는 CONNECTING으로 고정되고 이후 모든 connect/disconnect도 같은 락을 기다리며 막힌다.

**근거**
```
handleNetworkStateChange: `connectionStateMutex.withLock { ... attemptAutoReconnect(cameraToConnect) ... }` (370,405) → attemptAutoReconnect: `if (connectToCamera(camera, forceApMode = false))` (528) → connectToCamera: `connectionStateMutex.withLock { withContext(ioDispatcher) { ...` (691-693)
```

**수정 방향**
자동 재연결 경로는 락 밖에서 connect를 호출하도록 분리하거나, 락이 필요 없는 connectToCameraInternal(이미 락 보유 가정)을 만들어 호출. handleNetworkStateChange에서 락을 잡은 채 connect를 트리거하지 말고 상태 스냅샷만 취한 뒤 락 해제 후 재연결 코루틴을 launch.

**검증 근거**
전체 파일을 읽고 호출 경로를 추적해 데드락을 확인했다. handleNetworkStateChange(368-430)는 coroutineScope.launch(ioDispatcher) 안에서 connectionStateMutex.withLock(370)을 획득한 채, 같은 코루틴에서 attemptAutoReconnect(cameraToConnect)(405) 및 attemptAutoReconnect(updatedCamera)(423)를 suspend 호출한다. attemptAutoReconnect(507-559)는 별도 코루틴을 launch하지 않고 connectToCamera(camera, forceApMode=false)(528)를 직접 호출한다. connectToCamera(691-872)는 다시 connectionStateMutex.withLock(692)으로 동일 Mutex를 재획득하려 한다. kotlinx.coroutines.sync.Mutex(import 59-60)는 비재진입이므로, 락을 이미 보유한 동일 코루틴이 같은 락을 다시 withLock하면 영구 대기에 빠진다. 그 코루틴은 _connectionState=CONNECTING(525)으로 고정된 채 멈추고, 락을 영원히 쥐고 있으므로 disconnect(1367), 외부 connectToCamera(692), 이후 handleNetworkStateChange(370) 등 connectionStateMutex를 쓰는 모든 경로가 함께 차단된다. reconnectMutex(95)는 별개 Mutex로 동시 재연결만 막을 뿐 이 재진입을 방지하지 못한다. 진입 조건도 실제적이다: isAutoReconnectEnabled가 켜진 상태(setAutoReconnectEnabled로 사용자 설정)에서 Wi-Fi 재연결 시 branch1(387-410, lastConnectedCamera는 restoreLastConnectedCamera(328)로 복원됨) 또는 AP IP 변경 시 branch2(413-426)에 도달하면 결정론적으로 데드락된다. 가짜 양성 가능성을 검토했으나(다른 코루틴에서 launch되어 별개 코루틴이면 단순 대기일 뿐), attemptAutoReconnect와 connectToCamera 모두 launch 없이 suspend 직접 호출이라 동일 코루틴 컨텍스트가 유지됨을 확인했다. 라인 번호도 정확하다.

### 전역 eventListenerThread를 여러 경로가 보호 없이 join/detach — std::thread race로 UB/double-join

- **파일**: `app/src/main/cpp/camera_events.cpp:24, 1114-1167, 1257-1291; camera_ptpip.cpp 171-173, 411-413`
- **분류**: race | **차원**: C++ 이벤트/큐 | **확신**: medium

**문제**
전역 std::thread eventListenerThread가 최소 4개 경로에서 동기화 없이 접근된다. (1) stopListenCameraEvents가 joinWorker 스레드를 만들어 join, 타임아웃 시 joinWorker를 detach. (2) stopListenCameraEventsAsync가 별도 detached 스레드에서 eventListenerThread.joinable()/join()/detach()를 호출. (3) camera_ptpip.cpp의 initCameraWithPtpip와 initCameraForAPMode가 같은 객체를 직접 join. std::thread의 join()/detach()/joinable()/소멸자는 thread-safe하지 않다. stopListenCameraEvents(타임아웃→joinWorker가 백그라운드 join 진행)와 동시에 stopListenCameraEventsAsync 또는 initCameraWithPtpip가 같은 eventListenerThread.join()을 호출하면 동일 thread를 두 곳에서 join → std::system_error 또는 UB(크래시). detach 후 다른 곳이 join하면 invalid handle 접근.

**근거**
```
std::thread eventListenerThread;  // line 24, 보호 mutex 없음\n// stopListenCameraEvents: joinWorker.detach(); (타임아웃 시 백그라운드 join 지속)\n// stopListenCameraEventsAsync: if (eventListenerThread.joinable()) { eventListenerThread.join(); } ... eventListenerThread.detach();\n// camera_ptpip.cpp: if (eventListenerThread.joinable()) { eventListenerThread.join(); }
```

**수정 방향**
eventListenerThread 접근(join/detach/joinable/대입)을 전용 mutex로 직렬화하거나, 단일 종료 경로(예: stop 함수 하나)로 통합하고 나머지는 그 함수를 호출하도록 한다. 종료 후 std::thread를 비운 상태(default-constructed)로 명확히 리셋하여 중복 join을 원천 차단.

### stopListenCameraEventsAsync의 joinable() 폴링 + 무방비 join/detach 로직 결함

- **파일**: `app/src/main/cpp/camera_events.cpp:1257-1291`
- **분류**: race | **차원**: C++ 이벤트/큐 | **확신**: medium

**문제**
async 종료 스레드는 eventListenerThread.joinable() 루프로 대기하지만, joinable()은 join/detach 전까지 항상 true이므로 이벤트 스레드가 끝나도 자동으로 false가 되지 않는다. 루프는 eventListenerRunning==false면 break 후 join을 호출하는데, 이때 stopListenCameraEvents의 joinWorker(타임아웃 후 detach되어 백그라운드에서 같은 thread를 join 중)와 경쟁한다. 또한 타임아웃 catch 경로에서 eventListenerThread.detach()를 호출하는데, 이 detach가 다른 join 진행과 겹치면 UB. 두 종료 API가 동시에 호출될 수 있는 구조(콜백 기반 stop vs 동기 stop)에서 방어가 전혀 없다.

**근거**
```
while (eventListenerThread.joinable() && (now - start) < timeout) { sleep(100); if (!eventListenerRunning.load()) break; }\nif (eventListenerThread.joinable()) { eventListenerThread.join(); }\n... catch(...) { if (eventListenerThread.joinable()) eventListenerThread.detach(); }
```

**수정 방향**
단일 종료 경로로 통합하고 eventListenerThread 접근을 mutex로 보호. joinable() 폴링 대신 이벤트 스레드 종료를 알리는 condition_variable/atomic done 플래그로 대기.

handleNativePhotoDownload / handlePhotoDownload는 currentTier를 조회하지만 'FREE && JPEG'일 때 2000px 리사이즈만 적용한다(라인 413, 654). FREE 티어가 RAW나 PNG를 받으면 리사이즈/차단 없이 원본이 그대로 저장된다(FREE는 JPG/JPEG만 허용, 2000px 제한이 규약). 또한 BASIC 티어는 JPG/JPEG/PNG만 허용인데 RAW가 와도 어떤 게이팅도 적용되지 않는다. ValidateImageFormatUseCase가 단일 지점이어야 하나 이 매니저는 SubscriptionUtils/ValidateImageFormatUseCase를 전혀 호출하지 않고 getSubscriptionUseCase.getSubscriptionTier()만 읽어 자체 분기를 한다. 결과적으로 게이팅이 이벤트 콜백 한 곳에만 의존하게 되어, 위 finding처럼 콜백을 우회하는 경로에서 다운로드 제한·포맷 제한이 모두 사라진다.

**근거**
```
if (currentTier == SubscriptionTier.FREE && extension in Constants.ImageProcessing.JPEG_EXTENSIONS) {
    ... resizeImageForFreeTier(...)
}
// BASIC 분기 없음, RAW(extension in RAW_EXTENSIONS) 차단 없음, FREE의 PNG 처리 없음
```

**수정 방향**
다운로드 직전 ValidateImageFormatUseCase.resolveRawAccess()/isFormatSupported()로 포맷 허용 여부를 단일 지점에서 판정하고, 미허용 시 저장을 중단·콜백으로 알린다.
